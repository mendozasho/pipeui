## Plan (written before coding)

Sub-issues, climbing the layers: #238 (data + business) → #239 (business) →

- **Approach per layer**
  - **data (#238).** New `output_target_map` table in `schema/queries.py` DDL,
    keyed `(source_function_map_id, function_id)` with ordered
    `(column_id, position)` rows — the first table that maps a function's *output*
    to columns (ADR-0001 rejected re-using `alias_map`; alias_map is param-keyed
    *input*). DDL only on a fresh create (no production data, ADR consequence).
  - **business — attach (#238/#240).** `attach_function` gains an
    `output_targets: list[uuid.UUID] | None` param. For a `replace` step with
    targets provided, the target count MUST equal the bundle count (N from
    `pair_bundles` over the bound varying params); mismatch → structured failure
    (`ok=False`) BEFORE the write, never a 500. On success, write
    `output_target_map` rows in add-order *inside the same attach transaction*.
    The runner reads them ORDER BY position.
  - **business — run.py transform path (#239/#240).** Replace the single-column
    `_execute_transform_step` with a bundle loop mirroring the validation bundle
    loop (slice 3): pair the transform fn's bound columns into N argument bundles
    via `pair_bundles`, run the fn once per bundle, and apply `output_mode`:
    - **append** → add N new columns, each named by a cleaned auto-label
      (`normalize_label` of the varying column) or the user-provided append name;
      collisions de-duped with a numeric suffix so no append ever clobbers.
    - **replace** → overwrite the ordered target columns read from
      `output_target_map` (bundle i → target i); falls back to the input varying
      column when no explicit target (single-varying default, per PRD).
    - **pd.DataFrame** return → whole table in ONE run regardless of output_mode
      (no bundle expansion; existing `pd.dataframe` branch preserved).
    Each transform run yields a `RunResult` (one per bundle), labelled by its
    varying column.
  - **ui (#241).** In `PendingStepCard` (the mapping/binding card), add a
    replace-target column picker (ordered, one per bound column) shown when the
    step's output_mode is `replace`, and an optional append-name text input shown
    for `append`. Both follow the existing column-selector / scalar-input
    conventions and flow through `onSave` → the attach POST body
    (`output_targets`, `append_name`).
- **Utilities/patterns reused (file + symbol)**

## Actual (written before push)
- **What was built:** A new `output_target_map` table (DDL) and `attach_function`
  writes ordered replace-target rows (with a count==bundle-count guard, structured
  failure, atomic). The `run.py` transform path was rewritten into a bundle loop:
  append adds N collision-suffixed columns (cleaned function-name label or a
  user-provided name), replace overwrites the ordered targets read from
  `output_target_map` (bundle i → target i; single-varying defaults to the input
  column), and `pd.DataFrame` transforms run whole-table once. Each transform run
  yields a `RunResult`. The API forwards `output_targets`/`append_name`; the mapping
  modal gained an output-mode selector, a replace-target picker, and an append-name
  input flowing through `onSave`'s new `extras` arg.
- **actual_touched_files:**

  - src/pipeui/workflow/run.py
  - src/pipeui/api/pipelines.py

  - tests/test_api_pipelines.py
  - src/pipeui/frontend/screen-builder.test.jsx
  - .claude/runs/runner-execution/4.md
- **Deviations from plan (and why):**
  - `src/pipeui/workflow/bundles.py` and `src/pipeui/results.py` were planned but
    NOT modified — the existing `pair_bundles`/`ArgumentBundle` and
    `RunResult`/`normalize_label` from slices 3/1 covered the transform path as-is.
    Fewer files touched than predicted.
  - `tests/test_staging.py` was planned but NOT modified — it covers `CreateFlowCache`,
    not the runner staging table. The "transformed report reflects the transform"
    guarantee (#240 AC) is asserted at the higher run_workflow seam (per PRD line 81:
    "test_run_workflow.py = highest seam") by reading the written staging table.
  - The user-provided **append name is accepted by the API but not persisted** in v1
    (persisting it would add an `append_name` column to `source_function_map`, a
    19-call-site positional-INSERT fan-out across 7 test files — deferred as a
    non-load-bearing enhancement; the cleaned auto-label path fully meets AC #0).
    The runner reads `step.get("append_name")` (always None today) so the path is
    forward-compatible.
- **Unanticipated files (in actual but not planned or issue touched_files):** none.
