## Plan (written before coding)

### Approach per layer

**business_logic — `src/pipeui/results.py` (NEW) [#226, slice acceptance #5, #0, #1]**
- `normalize_label(raw: str) -> str` — pure function. Strips leading underscores and "odd
  tokens" (collapses runs of non-alphanumeric chars to single `_`, strips leading/trailing
  `_`), and never returns empty (falls back to a stable placeholder like `"result"`). Unit-
  tested directly in `tests/test_results.py`.
- `RunResult` — frozen dataclass (focused result-holder, per CONTEXT.md / PRD). Fields:
  `function_name`, `function_type` ("validation"|"transform"), `source_id`, `label`,
  `status`, plus a `bundle_key` (string identifying the argument bundle — for N=1 this is
  the single bound column name or ""). Identity `result_id` = shortened
  `UUID5(function, argument bundle, source)` computed via `pipeui.ids.content_hash_id`
  (reuse the existing two-level uuid5 factory — table_name namespace `"run_result"`,
  fields = function_name, bundle_key, source_id). Deterministic: equal inputs -> equal id.
  `.to_dict()` emits `result_id`, `label`, `function_type`, `status` (+ subclass fields).
- `ValidationRunResult(RunResult)` — adds `rows_passed`, `rows_failed`, `pass_rate`,
  `failing_rows`. `.to_dict()` extends base with those keys. This is the validation-
  specialized subclass the slice calls for.

**business_logic — `src/pipeui/workflow/run.py` (MODIFY) [#227, slice acceptance #0, #1]**
- Introduce a stateful per-step executor object (`_StepExecutor`) that holds the step's
  function, its input bindings, conn/source_id, and produces `RunResult`s — replacing the
  free-function `_execute_validation_step`/`_execute_transform_step` internals while keeping
  the N=1 column-dispatch logic identical (regression lock). The validation path builds a
  `ValidationRunResult` per function; the transform path builds a `RunResult` per step.
- `run_pipeline` serializes the produced RunResults into the step dicts via `.to_dict()`,
  preserving every existing wire key (status, function_type, rows_passed/failed, pass_rate,
  failing_rows, error, source_function_map_id, set_name, rows_affected) and ADDITIVELY
  including `result_id` and `label`. Backward-compatible: existing tests keep passing.
- Label for a result = `normalize_label(bound_column_name or function_name)`.

**api — `src/pipeui/api/pipelines.py`, `src/pipeui/api/validations.py` (MODIFY) [#228, acceptance #2]**
- No new endpoints. Because `run_pipeline` / `run_validation_across_sources` now emit
  RunResult dicts (with `result_id` + `label`), the existing `POST /pipelines/{id}/run`
  and `POST /validations/run` responses already carry the serialized RunResult fields.
  Verify at the API seam (TestClient) that each step/source result carries `result_id`
  (UUID5) and a readable `label`. Add per-source `label`/`result_id` to
  `run_validation_across_sources` source entries so the validations endpoint surfaces them.

**ui — `src/pipeui/frontend/screen-results.jsx`, `screen-data.jsx` (MODIFY) [#229, acceptance #3, #4]**
- screen-data.jsx `handleRunSource`: card_type per result is already split by
  `function_type` (validation/transform). #193 card-type fix = drive each card's type from
  the per-result `function_type` now carried on every result dict (RunResult), and surface
  the RunResult `label` + `result_id` on the card. Render one card per RunResult.
- screen-results.jsx: render label + counts; ensure mixed validation/transform set shows the
  correct `TypeTag` per card (driven by `card.card_type` <- result.function_type). Add named
  exports (`ScreenResults`, `ResultCard`, `SummaryLine`, `TypeTag`) for the vitest harness,
  mirroring screen-builder.jsx's window-global + named-export dual pattern.
