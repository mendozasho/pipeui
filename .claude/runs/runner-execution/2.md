# Slice 2 — Column order: alias_map.position + add-order + minimal reorder control
Issue: #219   Branch: feature/runner-execution-2

## Plan (written before coding)

Sub-issues, worked in order: #230 (data) → #231 (business) → #232 (api) → #233 (ui).

- **Approach per layer:**
  - **data (#230):** add `position INTEGER NOT NULL DEFAULT 0` to the `alias_map`
    table in `schema/queries.py` DDL. Breaking recreate is safe (no prod data).
  - **business (#231):**
    - `attach.py::attach_function` — when writing alias_map rows, `enumerate` the
      ordered `binding_map.get(p_id, [])` and write the index as `position`. The
      binding list already preserves add-order (built from `AttachBinding.column_ids`).
    - `attach.py::get_pipeline` — replace the `ORDER BY cr.column_name` in the
      per-param binding lookup with `ORDER BY am.position`.
    - `run.py::_fetch_steps` — the param/binding query LEFT JOINs alias_map and
      currently `ORDER BY p.param_name, cr.column_name`; change the column-order
      key to `am.position` (keep `p.param_name` as the param grouping key) so
      `bindings` lists are built in position order. (#193's N=1 path is unaffected
      — it reads `bindings[0]`, which is now position 0.)
  - **api (#232):** `attach.py::patch_pipeline_step` already replaces alias_map rows
    from `bindings: dict[param_id -> [col_id, ...]]`; `enumerate` the col_id list and
    write `position` on insert. The API route + `PatchStepIn.bindings` already carry
    the ordered list — no route signature change needed; the order in the JSON array
    is the position. Add an API test that PATCHes a new column order and reads it back
    via GET (`get_pipeline` bindings order).
  - **ui (#233):** in `screen-builder.jsx::PendingStepCard`, the selected column order
    IS `selections[p.param_id]` (an ordered array; `handleColumnToggle` appends on
    select). That array already flows to `onSave` → bindings → POST/PATCH, so order
    already persists once the backend honors it. Add a minimal move-up / move-down
    reorder control over the *selected* columns for a multi-column param, mutating the
    `selections[p.param_id]` array order. Frontend test: render a multi-column param
    with two selected columns, click "move down" on the first, click Save, assert
    `onSave` received the reordered `column_ids`.

- **Utilities/patterns reused (file + symbol):**

    already carry the ordered `bindings` list and pass it through to
    `patch_pipeline_step`; the position write lives entirely in the workflow layer.
    Confirmed by the green API test reading the new order back through GET. The
    PRD/issue named pipelines.py as a likely touch; it proved unnecessary.
  - `run.py` was named in the slice `touched_files`; the change there is the
    `ORDER BY` swap in `_fetch_steps` only (in-plan).
- **Unanticipated files (in actual but not planned or issue touched_files):**
  `tests/test_api_pipelines_run.py`, `tests/test_api_run_all.py`,
  `tests/test_api_run_set.py`, `tests/test_api_validations.py`. Cause: adding a 5th
  column to the shared `alias_map` table broke pre-existing test fixtures that used
  positional `INSERT INTO alias_map VALUES (?, ?, ?, ?)` (DuckDB binder error: 5
  columns, 4 values). Mechanical fix — converted each to the explicit-column form
  `INSERT INTO alias_map (alias_map_id, column_id, parameter_id, source_id) VALUES (...)`
  so the new `position` takes its DEFAULT. No production code in these paths. This is
  Phase-3 feedback: a DDL change to a shared table fans out to every positional
  fixture; the predicted `touched_files` for slice 2 should include the test files
  that seed `alias_map` directly.
