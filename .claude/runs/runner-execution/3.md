# Slice 3 — Multi-select validation execution (argument bundles)

## Plan (written before coding)

### Approach per layer

**business_logic — pure pairing (#234, slice acceptance #0/#1)**
- New pure module `src/pipeui/workflow/bundles.py` (no DB, no worker — `unit` only).
  - `pair_bundles(params)` where `params` is an ordered list of
    `{ "param_id"/"param_name", "columns": [col, ...] }` (columns already in
    user-placed `position` order, as `_fetch_steps` emits via `ORDER BY am.position`).
  - Classify each param: **static** (len==1, broadcast) vs **varying** (len>1) vs
    empty (len==0 → no expansion contribution; treated as N=1 single-col path).
  - Equal-length-among-varying rule: all *varying* params must share one length `N`.
    Distinct lengths among varying params → raise `BundleLengthError` (a `ValueError`
    subclass) naming the conflicting counts. Length-1 broadcasts. N is the bundle count.
  - All-static / single-param / no varying → N = max(1, single varying len). Matrix:
    `3,3,1`→3 (broadcast the 1), single-param `N`→N, all-static→1, unequal `3,2`→raise.
  - Returns an ordered list of bundles; bundle `i` = each varying param's `i`-th column +
    every static param's single column. A bundle is `{param_id: column, ...}` plus a
    `varying` label list for result labeling.
- Wire backend reject into `attach.py`: after binding collection, call the pure
  pairing fn over the varying/static column counts; on `BundleLengthError` return the
  existing **structured failure** dict (`{"ok": False, "missing_params": [], "detail": ...}`)
  — NOT a raised 500. Sits next to the `position` writes from slice 2.

**business_logic — executor bundle loop (#235, slice acceptance #2/#3)**
- In `run.py` `_execute_validation_step`: for a `multi_select_eligible` validation
  function (column-backed/pd.Series param with bound columns), build bundles via
  `pair_bundles` and run the function **once per bundle**, emitting one
  `ValidationRunResult` per bundle, each labeled by its varying column(s) (normalized).
  N=1 path (single bound column) stays one result, unchanged behavior.
- `scalar run` normalization: a scalar-shaped validation (str-not-column_backed / int /
  bool param, already dispatched via the `.apply()` wrapper) runs once per record and
  yields a normalized boolean vector — already the case for the str-scalar path; ensure
  the bundle loop drives the per-column dispatch and the per-row vector is collected
  and `.astype(bool)`-normalized (existing interpretation block).
- Refactor: extract a `_run_one_validation_bundle(...)` helper from the existing
  per-function body so the bundle loop calls it per bundle (own-diff refactor only).

**api (#236, slice acceptance #4)**
- `POST /pipelines/{source_id}/run` already serializes `result["steps"]`, one dict per
  emitted `ValidationRunResult.to_dict()`. With the executor now emitting N per bundle,
  the endpoint returns N per-bundle RunResults with no route change — assert via
  TestClient that a function bound to N columns yields N step dicts with distinct labels
  and result_ids. (The Results screen already renders one card per RunResult — slice 1 —
  so the N-cards UI behavior is covered by the existing screen-results render path.)
- Equal-length violation surfaces as a structured failure at **attach** (#234), so the
  API test for the reject lives against `POST /steps` (test_api_validations.py /
  test_api_pipelines.py): a mismatched multi-bind attach returns ok=False / 4xx, not 500.

**ui (#237, slice acceptance #5)**
- The mapping modal (`PendingStepCard` in `screen-builder.jsx`) already binds multiple
  columns to one param (multi-select `selections[param_id]` array + reorder, slice 2).
  Add a vitest test asserting multi-column bind → onSave sends a `column_ids` array of
  length >1 for one param (existing convention; mostly a coverage/guard test).
- Frontend equal-length block: when two+ varying params (each >1 selected column) have
  different selected-column counts, disable Save and show an inline mismatch message —
  mirrors the backend guard so the user is blocked before the POST. Pure render-state
  logic in `PendingStepCard`; tested via vitest.

## Follow-ups / debt
- **Multi-param-per-bundle execution** (e.g. one function with `country` × `team`, two
  eligible params paired positionally) is deferred: the worker `call_function` passes a
  single kwarg, so the executor currently expands the single eligible column param. The
  pure `pair_bundles` already supports multi-param pairing — the missing piece is a
  multi-kwarg worker dispatch. Out of scope for this slice's acceptance.
- **Test archive-vs-promote recommendation (per `.claude/conventions.md` archive default):**
  - `tests/test_bundle_pairing.py` → **PROMOTE.** Pure exhaustive coverage of the
    load-bearing pairing seam; asserts stable behavior (zip/broadcast/equal-length), not
    phrasing. The PRD names this the load-bearing seam — keep it in the permanent suite.
  - `tests/test_run_workflow.py` (new bundle-loop cases) → **PROMOTE.** They lock the
    one-RunResult-per-bundle and scalar-run guarantees against future executor edits.
  - `tests/test_attach.py` (equal-length reject cases) → **PROMOTE.** Locks the
    structured-failure / no-partial-write attach guarantee.
  - `tests/test_api_pipelines_run.py` + `tests/test_api_pipelines.py` (new cases) →
    **ARCHIVE.** API-seam integration evidence of the same guarantees the workflow/pure
    tests already lock permanently; the red-green having run is the evidence.
  - `screen-builder.test.jsx` (new cases) → **PROMOTE** the equal-length-block + multi-bind
    cases (stable UI contract mirroring the backend guard); they are not phrasing-bound.
