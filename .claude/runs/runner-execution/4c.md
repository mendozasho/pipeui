# Slice 4c — Registry-migration fix for pre-feature DBs (#254)
Issue: #254   Branch: feature/runner-execution-4c

to-code addendum (NOT in the frozen slice ledger). Bug found during manual testing of PR #253.

## Plan
- Root cause: slices 2/4b added `alias_map.position` and `source_function_map.append_name` to the DDL but never registered them in `db._REGISTRY_SCHEMA_MIGRATIONS`, so existing DBs (created before this feature) keep the old shape and `get_pipeline` 500s on `ORDER BY am.position`.
- Fix: add the two columns to `_REGISTRY_SCHEMA_MIGRATIONS` (mechanism already exists, idempotent ALTER-or-rollback).
- Test command: `PYTHONPATH=src <venv> -m pytest tests/test_schema.py`
- planned_touched_files: [src/pipeui/db.py, tests/test_schema.py, .claude/runs/runner-execution/4c.md]

## Actual
- Built: registered `("alias_map","position","INTEGER DEFAULT 0")` and `("source_function_map","append_name","VARCHAR")` in `_REGISTRY_SCHEMA_MIGRATIONS`. Existing DBs self-migrate on startup.
- actual_touched_files: src/pipeui/db.py, tests/test_schema.py, .claude/runs/runner-execution/4c.md
- Deviation: dropped `NOT NULL` from the position migration — DuckDB rejects `ADD COLUMN` with constraints ("Adding columns with constraints not yet supported"). `DEFAULT 0` backfills existing rows; inserts always supply position, so no NULL arises. Matches the existing `is_active` entry's constraint-free style.
- Unanticipated files: none.
