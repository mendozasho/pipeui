# Slice 5 — Results-screen consumption: exports (results report + transformed report) + minimal drawer
Issue: #222 (sub-issues #242, #243, #244)   Branch: feature/runner-execution-5

## Plan (written before coding)

### Approach per layer

**business_logic — `src/pipeui/workflow/export.py` (new):**
- `build_results_report(run_result) -> list[dict]` — the **transposed results report**: one row
  per RunResult keyed by its normalized `label`, with pass/fail + metadata columns. Accepts either
  shape the runner produces:
  - the source-tied validation run shape `run_pipeline(..., "validations")` → `{ run_type, steps: [...] }`
    where each step dict already carries the RunResult contract (`result_id`, `label`,
    `function_name`, `function_type`, `status`, `rows_passed`, `rows_failed`, `pass_rate`).
  - the Functions-page cross-source shape `run_validation_across_sources(...)` →
    `{ function_id, function_name, sources: [...] }` where each per-source entry IS a RunResult of
    that function on a source (`result_id`, `label`, `status`, `rows_passed`, `rows_failed`, `pass_rate`).
  Both entry points → the same transposed row list. **Includes runs that fully passed** (no filtering
  on rows_failed). Labels are already normalized by the runner via `normalize_label`; the export
  re-normalizes the `label` column defensively (acceptance #3 — no leading underscores/odd tokens)
  by reusing `results.normalize_label` (read-only import — results.py is not modified).
- `build_transformed_report(conn, source_id) -> {columns, rows}` — the **transformed report**: the
  transformed data table after all assigned transforms completed. Reuses the existing
  `workflow.run.get_staging_rows` (the latest staging table = transformed report). This is the
  read that backs the transformed export and is the path that must not fail for a mixed
  validation/transform set (#193 export symptom): a validation-only run writes no staging table, so
  the function returns `{columns: [], rows: []}` rather than raising.

**api — `src/pipeui/api/pipelines.py` + `src/pipeui/api/validations.py`:**
- `GET /pipelines/{source_id}/export/transformed` → `build_transformed_report` ({columns, rows}); 404
  unknown source. (transformed report — owns #243 / slice #2, #4)
- `GET /pipelines/{source_id}/export/results?run_type=validations` → runs the source-tied validation
  pipeline and returns `build_results_report` of it (transposed; source-tied entry point). 404 unknown
  source. (results report entry point 2 — owns #242 / slice #0, #1)
- `GET /validations/{function_id}/export/results` → runs `run_validation_across_sources` and returns
  `build_results_report` of it (transposed; Functions-page entry point). 404 unknown function.
  (results report entry point 1 — owns #242 / slice #0, #1)
  Each row carries pass/fail/metadata incl. passing runs; labels normalized.

**ui — `src/pipeui/frontend/screen-results.jsx` + `ui.jsx` (reuse `Drawer`):**
- Minimal results drawer: clicking a result card opens the EXISTING `Drawer` component
  (`window.__UI__.Drawer`) rendering the RunResult metadata (label, function_name, function_type,
  status, result_id, counts). No new drawer authored — AC #5 says reuse. (owns #244 / slice #5)
- Export buttons already exist on the card; the slice's net-new UI guarantee is the drawer.
