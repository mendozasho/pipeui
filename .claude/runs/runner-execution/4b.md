# Slice 4b — Persist user-provided append name server-side (append_name on source_function_map)
Issue: #251   Branch: feature/runner-execution-4b

A to-code addendum (not in the frozen slice ledger), closing the persistence gap
left as deferred debt in slice 4's build record. The runtime in `run.py` ALREADY
consumes `step.get("append_name")` (run.py:304, :390, :449); this slice closes the
persistence plumbing only — it does NOT rewrite naming logic.

## Plan (written before coding)

Verified facts (before coding):
- `run.py:304` reads `append_name = step.get("append_name")`; :390/:449 use it as
  `append_name or normalize_label(fn_name)`. Consumption is intact.
- `source_function_map` DDL (queries.py:53) has columns
  (source_function_map_id, source_id, set_id, position, output_mode) — NO append_name.
- `attach_function` (attach.py:235) does a 5-col INSERT; no append_name param.
- `_fetch_steps` (run.py:143-160) SELECTs 5 cols, never append_name → step["append_name"] is always None.
- API request model `_AttachBody` (pipelines.py:45) HAS `append_name` but the
  `attach_function(...)` call (pipelines.py:189) does NOT forward it.
- 20 positional `INSERT INTO source_function_map VALUES (?, ?, ?, ?, ?)` fixtures
  across 7 test files (test_api_run_set, test_run_workflow, test_api_validations,
  test_api_run_all, test_ingestion, test_api_pipelines_run, test_functions).

Approach per layer:
- **data (schema/queries.py).** Add `append_name VARCHAR` (nullable, no default) to
  the `source_function_map` DDL. Pure additive on a fresh-create schema (no prod data).
- **business — attach.py.** Add `append_name: str | None = None` kwarg. Persist it in
  the INSERT (explicit-column, 6 values) inside the existing attach transaction. Per
  the ticket, persist when `output_mode == "append"`; for replace we store NULL.
- **business — run.py `_fetch_steps`.** SELECT `sfm.append_name`, unpack it, set
  `step["append_name"]` so the existing naming logic fires. (No change to the naming
  loop itself — it already reads the key.)
- **api — api/pipelines.py.** Forward `append_name=body.append_name` in the
  `attach_function(...)` call.
- **fixtures.** Convert all 20 positional `INSERT INTO source_function_map VALUES (...)`
  to explicit-column inserts naming the original 5 columns, so the new column defaults.
  Same pattern slice 2 used for `alias_map.position` / slice 4 for output_target_map.

Utilities/patterns reused:
- Existing `attach_function` transaction + INSERT pattern (slices 2/4).
- Existing `_fetch_steps` SELECT/unpack shape.
- Existing run.py naming logic (`append_name or normalize_label(fn_name)`, collision

- `normalize_label` (slice 1) — already applied by run.py to clean the name.
- Test helpers in test_attach.py (`_make_function`), test_run_workflow.py

## Actual (written before push)
- **What was built:** Closed the append_name persistence plumbing exactly as scoped —
  no naming-logic rewrite. (1) Added `append_name VARCHAR` (nullable) to the
  `source_function_map` DDL. (2) `attach_function` gained an `append_name` kwarg; it
  is **normalized via `normalize_label` at persistence time** (so the stored value is
  already a clean column name) and written only for `output_mode == "append"` with a
  non-blank name — NULL otherwise. The existing run.py consumption uses the value
  verbatim, so normalizing at attach satisfies AC #2's "cleaned/normalized" without
  touching the run.py naming loop. (3) `_fetch_steps` now SELECTs `sfm.append_name`
  and sets `step["append_name"]`, feeding the existing run.py consumer (run.py:304/449).
  (4) The API (`api/pipelines.py`) now forwards `body.append_name` into
  `attach_function` (the request model already carried it from slice 4 / #241). (5)
  Converted all 20 positional `INSERT INTO source_function_map VALUES (?, ?, ?, ?, ?)`
  fixtures (the slice-4b helper `_seed_transform_step` + 19 others across 7 test files)
  to explicit-column inserts naming the original 5 columns, so the new column defaults.
- **Design note (where normalization lives):** run.py consumes `append_name` raw
  (`append_name or normalize_label(fn_name)`). Per the slice instruction NOT to rewrite
  the naming logic, the user name is cleaned at the persistence boundary (attach) rather
  than in run.py. This is persistence shaping, not a naming-logic change.
