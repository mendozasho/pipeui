# Slice 1 — resolve_frame input-source seam
Issue: #11   Branch: feature/runner-resolution-model-1

## Plan (written before coding)
- Approach per layer:
  - **data / business_logic:** New module `src/pipeui/workflow/resolve.py` exposing
    `resolve_frame(conn, source_id, mode) -> (pd.DataFrame, FrameRef)`.
    - `mode="raw"` → read the source's instance table (reuse the
      `instance_table_name` helper); `FrameRef` carries `mode="raw"`, `source_id`,
      and `result_id=None` (raw is the source's own data, not a produced result).
    - `mode="transformed"` → find the source's latest staging table
      (`staging_{source_id8}_{ts}`, highest ts). If present, read it. If absent,
      **materialize-if-absent**: run the source's transforms pipeline once
      (`run_pipeline(conn, source_id, "transforms")`) then read the produced
      staging table. `FrameRef` carries a deterministic `result_id`.
  - **Cycle guard:** `resolve_frame` threads an `_in_progress: frozenset[uuid]` of
    source_ids currently being materialized. Before materializing a source, if it
    is already in `_in_progress` → raise `TransformedCycleError` naming the cycle
    sources. Materialization scans the source's built-in **join** steps that
    reference another source's *transformed* output (`use_transformed` true in the
    join config) and recurses `resolve_frame(..., transformed, _in_progress + {self})`
    first — this is the path a cycle forms through, and it is forward-compatible
    with slice 2 wiring the join. Does NOT change the join execution itself.
  - **Transformed-output identity (`results.py`):** add a small helper
    `transformed_result_id(source_id, mode, staging_ts)` reusing the same
    `content_hash_id` / `_RESULT_NAMESPACE` identity scheme as `RunResult`, so the
    transformed-output ref's `result_id` is deterministic and consistent with the
    RunResult identity scheme.
- Utilities/patterns reused:
  - `pipeui.sql_user_table.instance_table_name`
  - `pipeui.workflow.run._staging_prefix`, `run_pipeline`
  - `pipeui.results.content_hash_id` / `_RESULT_NAMESPACE` (RunResult identity helper)
  - `pipeui.workflow.builtins.get_builtin_steps` (to find transformed join refs for the cycle guard)
  - test helpers `create_source` + `ingest_source` (real instance tables), `make_csv`
- Test command (red-green cycles): `PYTHONPATH=src .venv/bin/pytest tests/test_run_workflow.py -q`
- planned_touched_files:
  - src/pipeui/workflow/resolve.py
  - src/pipeui/results.py
  - tests/test_run_workflow.py
  - .claude/runs/runner-resolution-model/1.md
  - (run.py: only if an import/helper export is needed — avoid changing if possible)

## Actual (written before push)
- What was built: New `workflow/resolve.py` with `resolve_frame(conn, source_id, mode)
  -> (pd.DataFrame, FrameRef)`. raw reads the instance table; transformed uses the
  latest staging table or materializes-if-absent (runs the source's transforms
  pipeline once), cycle-guarded via a threaded `_in_progress` frozenset that raises
  `TransformedCycleError` naming the cycle sources. Added `transformed_result_id`
  to `results.py` reusing the RunResult UUID5 identity scheme for the transformed
  ref's deterministic `result_id`.
- actual_touched_files:
  - src/pipeui/workflow/resolve.py (new)
  - src/pipeui/results.py
  - tests/test_run_workflow.py
  - .claude/runs/runner-resolution-model/1.md
- Deviations from plan (and why): Did NOT touch `src/pipeui/workflow/run.py` —
  the identity helper went in `results.py` and the latest-staging lookup is a small
  private helper local to `resolve.py` (avoids importing run.py at module load and
  the circular-import risk; `run_pipeline` is imported lazily inside `_materialize`).
  So run.py — a predicted touched_file — was NOT modified. This narrows the slice's
  footprint and reduces overlap with siblings 3/4/5 that also edit run.py.
- Unanticipated files: none (all actual files were in the issue's predicted
  touched_files; resolve.py was predicted, run.py was predicted but not needed).
