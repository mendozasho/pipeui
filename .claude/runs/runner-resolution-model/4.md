## Plan (written before coding)
- Approach per layer (business_logic only):
  - **executors.py:** introduce a `FunctionSetAdapter` that flattens a function
    set's members into a stream of per-member `StepContext`s and dispatches each
    through `STEP_EXECUTORS` by the member's own `step_type` (type-agnostic — not
    hardcoded to function). Each per-member sub-context carries a one-function
    `functions` list, so the EXISTING per-member function executor runs it exactly
    as it runs a single-function step today — behavior-preserving by construction.
    - Rename today's `FunctionStepExecutor` role: it stays the per-MEMBER function
      executor (runs one function's transforms+validations). A new
      `FunctionSetExecutor` (the adapter) is registered under the `FUNCTION`
      step-type key that `run_pipeline` already dispatches sets to, so the set is
      flattened transparently. Member dispatch resolves a per-member executor from
      the registry by the member's `step_type` (defaults to `function`), so a
      non-function member object routes elsewhere (heterogeneity readiness).
    - Staging/working threading: the adapter threads the `working` frame member to
      member and accumulates `entries`; members share the run's `ts`, so the final
      staging table (DROP IF EXISTS + CREATE on the same name) holds the fully
      chained frame — identical to the pre-refactor single-step write. Validations
      read `env.original_df` (unchanged).
  - **run.py:** no dispatch-loop change needed (it already calls
    `STEP_EXECUTORS[ctx.step_type]`); the only change is which executor sits under
    the `FUNCTION` key (now the adapter). Keep `_step_has` / `_execute_*` helpers.

## Actual (written before push)
- What was built: a `FunctionSetExecutor` adapter (executors.py) that flattens a
  function set's members into per-member single-member sub-contexts and dispatches
  each through `STEP_EXECUTORS` by the member's own step type (defaulting to
  `FUNCTION`), threading the working frame and concatenating entries. The registry
  is re-keyed: a function-map row is now a `SET` step (new `SET` constant in
  context.py; `from_set` tags it; `for_step` routes function-map rows to
  `from_set`) -> the adapter; the adapter re-dispatches each plain function member
  as `FUNCTION` -> the existing per-member `FunctionStepExecutor` (unchanged) ->
  the same transform/validation helpers. Behavior-preserving: members share the
  run timestamp so the final staging table holds the fully chained frame.
- actual_touched_files:
  - src/pipeui/workflow/context.py (add `SET`; `from_set` tags SET; `for_step`
    routes function-map rows to `from_set`)
  - src/pipeui/workflow/executors.py (add `FunctionSetExecutor`; re-key
    `STEP_EXECUTORS` SET->adapter, FUNCTION->per-member, BUILTIN->builtin)
  - tests/test_run_workflow.py (add 3 slice-4 tests + `_comparable_entry` helper)
  - .claude/runs/runner-resolution-model/4.md
- Deviations from plan (and why): the plan registered the adapter under the
  existing `FUNCTION` key, but a plain function member's step type is also
  `FUNCTION`, which would recurse (adapter -> adapter). Resolved by introducing a
  distinct `SET` step type for the function-map row (the adapter) and keeping
  `FUNCTION` for the per-member executor; the adapter dispatches members through
  the registry by member step type, so `FUNCTION` members reach the per-member
  executor and a non-function member reaches its own executor. context.py was
  touched (not in the plan) to add `SET` and adjust `from_set`/`for_step` — this
  is the routing change the adapter needs; behavior of existing function/builtin
  steps is unchanged (full suite green).
- Unanticipated files: src/pipeui/workflow/context.py is in neither the slice's
  predicted `touched_files` nor `planned_touched_files` for this slice (it was a
  slice-3 file). The `SET` step type + `from_set`/`for_step` routing live there by
  the module's design, so the adapter could not be wired without it. run.py (in
  the issue's predicted `touched_files`) needed NO change — the registry-driven
  dispatch loop + `setdefault("step_type","function")` already route function-map
  rows through `for_step` -> `SET` -> the adapter. Phase-3 feedback: predicted
  run.py, but the real edit was context.py.
