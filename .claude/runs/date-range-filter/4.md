## Plan (written before coding)
- Approach per layer (data / business / api / ui):
  - **ui only** — everything lives in `src/pipeui/frontend/screen-builder.jsx`.
    No backend changes: slice 2 seeded `date_range` in `builtin_registry` (so the
    registry-driven palette already lists it via `GET /builtins`), and slice 3 owns
    the attach/patch write boundary. This slice wires the browser side.
  - **DateRangeModal** (new component, after RenameModal): props
    `{open, onClose, currentSource, onSubmit, initialConfig}` — same create/edit
    contract as Filter/Rename. State: `groups = [{conditions:[{column,start,end}]}]`,
    seeded from `initialConfig.groups` on open (persisted order kept, Principle 7)
    or one empty condition. Column picker per condition offers only columns with
    `column_type ∈ {DATE, TIMESTAMP, TIMESTAMPTZ}` (matches backend
    `_DATE_COLUMN_TYPES`). Start/end are native `<input type="date">` (frozen
    decision — no CDN picker). `+ Add condition` (per group), `+ Add OR group`,
    per-row remove (removing a group's last row removes the group). Save disabled
    with a visible reason for: zero conditions, missing column, both bounds empty,
    start > end — mirroring `_validate_date_range_config`. Untouched bounds keep
    their seeded values (null stays null, "" stays "") so an untouched save
    re-emits an identical config.
  - **submitDateRangeStep** in SidePanel: mirrors `submitRenameStep` — POST
    `/sources/{id}/attach-builtin` with `builtin_type:"date_range"` on create,
    PATCH `/sources/{id}/attach-builtin/{step_id}` when `editingBuiltin` set.
  - **Drop routing** in `handleDrop`: `date_range` branch — if `pipeline.steps`
    already holds a `builtin_type === "date_range"` step (singleton), set
    `editingBuiltin` to it before opening the modal (drop-again-edits); else open
    empty. Note: no existing builtin has this drop-again routing today (rename's
    drop just opens an empty modal and relies on the backend singleton rejection),
    so this branch is new logic, not a copy.
  - **handleEditBuiltin**: add `date_range` branch (pencil on the placed card).
  - **builtinConfigSummary**: add `date_range` branch — single group+condition
    renders `Date range · <col> <start> → <end>` ("…" for an open bound);
    otherwise `Date range · N conditions · M groups`.
  - **Testability seams**: add `data-testid="pipeline-dropzone"` to SidePanel's
    drop div; export `DateRangeModal` and `RightPalette` alongside the existing
    named exports.
- Utilities/patterns reused (file + symbol):
  - `RenameModal` (screen-builder.jsx) — add/remove-row pattern, open/onClose/
    onSubmit/initialConfig contract, Principle-7 seed-on-open effect.
  - `submitRenameStep` / `handleEditBuiltin` / `handleDrop` builtin branches.
  - `builtinConfigSummary` + `BuiltinStepCard` (generic — renders any builtin).
  - `_filterFieldStyle` shared field style; `window.__UI__` `Modal`/`Btn`.
  - Test side: `LEFT_SOURCE`-style fixtures, `renderRename`-style helper,

### Criterion → test mapping (sub-issue #125 first: criteria 2–5; then #126: 1 and 6)
1. Palette card + drop routing (create vs edit) — RightPalette shows the Date Range
   card from a stubbed `/builtins` (regression guard; the registry-driven palette
   makes this green-by-construction — will note if no true red is observable);
   SidePanel drop with no date_range step opens an empty modal; drop with one
   attached opens it pre-filled and saves via PATCH.
2. Date-only column picker — modal column select offers only DATE/TIMESTAMP/
   TIMESTAMPTZ columns from `currentSource.columns`.
3. Add/remove — `+ Add condition` appends to the current group, `+ Add OR group`
   starts a new group, per-row remove deletes exactly that row; start/end inputs
   have `type="date"`.
4. Validation + payload — save blocked with visible reason for zero conditions /
   both bounds empty / start > end; valid save submits exactly
   `{groups:[{conditions:[{column,start,end}]}]}`.
5. Round-trip — open persisted (multi-group, non-trivial order), save untouched,
   `onSubmit` receives a deep-identical config in persisted order.
6. Card summary — BuiltinStepCard renders a readable date_range summary (single
   condition: column + bounds; multi: group/condition counts).

## Follow-ups / debt
- Rename's own drop path still opens an empty modal when a rename step already
  exists (the backend 422s the second attach). date_range now has the friendlier
  singleton drop-again-edits routing; porting it to rename is a candidate
  follow-up, deliberately out of this slice's scope.
- The four submit*Step functions in SidePanel (join/filter/rename/date_range) are
  structurally identical up to builtin_type + flash label — a parameterized
  `submitBuiltinStep(builtinType, label)` would collapse them. Not done here:
  three of the four are surrounding code, not this slice's diff.
- Test-evidence default: the 18 authored tests assert stable component behavior
  (not phrasing) and sit in the same permanent describes file as the existing 96 —
  recommend **promote** into the permanent suite.
