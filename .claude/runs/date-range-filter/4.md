## Plan (written before coding)
- Approach per layer (data / business / api / ui):
  - **ui only** ‚Äî everything lives in `src/pipeui/frontend/screen-builder.jsx`.
    No backend changes: slice 2 seeded `date_range` in `builtin_registry` (so the
    registry-driven palette already lists it via `GET /builtins`), and slice 3 owns
    the attach/patch write boundary. This slice wires the browser side.
  - **DateRangeModal** (new component, after RenameModal): props
    `{open, onClose, currentSource, onSubmit, initialConfig}` ‚Äî same create/edit
    contract as Filter/Rename. State: `groups = [{conditions:[{column,start,end}]}]`,
    seeded from `initialConfig.groups` on open (persisted order kept, Principle 7)
    or one empty condition. Column picker per condition offers only columns with
    `column_type ‚àà {DATE, TIMESTAMP, TIMESTAMPTZ}` (matches backend
    `_DATE_COLUMN_TYPES`). Start/end are native `<input type="date">` (frozen
    decision ‚Äî no CDN picker). `+ Add condition` (per group), `+ Add OR group`,
    per-row remove (removing a group's last row removes the group). Save disabled
    with a visible reason for: zero conditions, missing column, both bounds empty,
    start > end ‚Äî mirroring `_validate_date_range_config`. Untouched bounds keep
    their seeded values (null stays null, "" stays "") so an untouched save
    re-emits an identical config.
  - **submitDateRangeStep** in SidePanel: mirrors `submitRenameStep` ‚Äî POST
    `/sources/{id}/attach-builtin` with `builtin_type:"date_range"` on create,
    PATCH `/sources/{id}/attach-builtin/{step_id}` when `editingBuiltin` set.
  - **Drop routing** in `handleDrop`: `date_range` branch ‚Äî if `pipeline.steps`
    already holds a `builtin_type === "date_range"` step (singleton), set
    `editingBuiltin` to it before opening the modal (drop-again-edits); else open
    empty. Note: no existing builtin has this drop-again routing today (rename's
    drop just opens an empty modal and relies on the backend singleton rejection),
    so this branch is new logic, not a copy.
  - **handleEditBuiltin**: add `date_range` branch (pencil on the placed card).
  - **builtinConfigSummary**: add `date_range` branch ‚Äî single group+condition
    renders `Date range ¬∑ <col> <start> ‚Üí <end>` ("‚Ä¶" for an open bound);
    otherwise `Date range ¬∑ N conditions ¬∑ M groups`.
  - **Testability seams**: add `data-testid="pipeline-dropzone"` to SidePanel's
    drop div; export `DateRangeModal` and `RightPalette` alongside the existing

### Criterion ‚Üí test mapping (sub-issue #125 first: criteria 2‚Äì5; then #126: 1 and 6)
1. Palette card + drop routing (create vs edit) ‚Äî RightPalette shows the Date Range
   card from a stubbed `/builtins` (regression guard; the registry-driven palette
   makes this green-by-construction ‚Äî will note if no true red is observable);
   SidePanel drop with no date_range step opens an empty modal; drop with one
   attached opens it pre-filled and saves via PATCH.
2. Date-only column picker ‚Äî modal column select offers only DATE/TIMESTAMP/
   TIMESTAMPTZ columns from `currentSource.columns`.
3. Add/remove ‚Äî `+ Add condition` appends to the current group, `+ Add OR group`
   starts a new group, per-row remove deletes exactly that row; start/end inputs
   have `type="date"`.
4. Validation + payload ‚Äî save blocked with visible reason for zero conditions /
   both bounds empty / start > end; valid save submits exactly
   `{groups:[{conditions:[{column,start,end}]}]}`.
5. Round-trip ‚Äî open persisted (multi-group, non-trivial order), save untouched,
   `onSubmit` receives a deep-identical config in persisted order.
6. Card summary ‚Äî BuiltinStepCard renders a readable date_range summary (single
   condition: column + bounds; multi: group/condition counts).
