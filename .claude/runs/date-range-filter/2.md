## Plan (written before coding)
- Approach per layer (data / business / api / ui):
  - **data (#120):** add the `date_range` row to `SEED_BUILTINS` in
    `src/pipeui/backend/data/base/schema/queries.py` (builtin_id `...0005`,
    config_schema `{"groups": "array"}`). Pre-feature-DB migration follows the
    rename (#40) precedent: `SEED_BUILTINS` is `INSERT OR IGNORE` and re-runs on
    every `create_schema` call (app startup ŌĆö `middleware/deps.py`), so an existing
    DB gains the row on next start and re-running is idempotent (UNIQUE on
    `builtin_type`). No `_REGISTRY_SCHEMA_MIGRATIONS` entry needed ŌĆö no column
    change. Proven by test, not assumed.
  - **business (#120):** register `date_range` in `BUILTIN_EXECUTORS` as
    `BuiltinSpec(validate=_validate_date_range_config, execute=<adapter lambda>,
    singleton=True, pinned_tail=1)` ŌĆö slice 1's pinned-tail mechanism places it
    after all positional steps and before rename (rank 2) at all three ordering
    consumers with zero sort-site changes.
  - **business (#121):** pure validator `_validate_date_range_config(cfg) -> str | None`
    over `{"groups": [{"conditions": [{"column", "start", "end"}]}]}`. Rejects:
    zero groups, an empty group (no conditions), a condition missing a column,
    both bounds empty (None/""), start > end. Accepts one-sided conditions and
    multi-group configs. No DB access.
  - **business (#122):** executor `_execute_date_range(conn, df, cfg)` following
    `_execute_filter`'s temp-view + WHERE + bound-params pattern. One WHERE of
    OR-ed group predicates; each group an AND of inclusive range predicates
    `CAST("col" AS DATE) >= CAST(? AS DATE)` / `<= CAST(? AS DATE)` (only the
    present bounds). TIMESTAMP/TIMESTAMPTZ therefore compare at DATE granularity;
    NULL dates fail their condition by SQL semantics but the row can pass via
    another OR group.
  - api / ui: none in this slice (attach-boundary eligibility is slice 3; UI is slice 4).
