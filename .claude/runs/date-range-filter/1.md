## Plan (written before coding)
- Approach per layer (business_logic only):
  - **Metadata (one place):** add `pinned_tail: int | None = None` to the frozen
    `BuiltinSpec` dataclass in `src/pipeui/backend/domain/functions/builtins.py`.
    Contract: `None` = positional step (rank 0, sorts by stored position);
    an `int >= 1` = pinned-tail rank — the step sorts after every positional step,
    ordered among pinned steps by rank (lower runs earlier). `rename` registers
    `pinned_tail=2`, reserving rank 1 for the `date_range` step slice 2 adds
    (PRD tail order: [positional steps..., date_range, rename]).
  - **One shared rank helper:** `pinned_tail_rank(builtin_type: str | None) -> int`
    in `builtins.py`, reading `BUILTIN_EXECUTORS` at call time (so a spec
    registered/overridden in a test is honored). Function steps and unknown /
    unpinned builtin types rank 0.
  - **Rewrite the three sort sites to consume it** (no literal "rename" anywhere
    in an ordering key):
    1. `get_unified_pipeline` in `builtins.py` (unified pipeline payload)
    2. `get_pipeline` in `src/pipeui/backend/domain/functions/pipeline_read.py`
       (pipeline read payload) — imports the helper from `builtins.py`
       (safe: `builtins.py` does not import `pipeline_read`)
    3. `run_pipeline`'s `active_steps` sort in
       `src/pipeui/backend/domain/runner/run.py` (execution order) — imports the
       helper (safe: `domain/runner/executors.py` already imports from
       `builtins.py`, so this direction exists today)
  - Retire the `NOTE (#83)` comment at run.py — this slice is that promotion.

### Red-green mapping per acceptance criterion
1. **Regression lock (rename sorts identically at all three consumers).**
   Existing tests `test_rename_pinned_last_in_pipeline_display` (get_pipeline)
   and `test_rename_pinned_last_in_execution` (run_pipeline) stay green
   throughout. The third consumer has no existing lock — author
   `test_rename_pinned_last_in_unified_pipeline` (get_unified_pipeline) BEFORE
   the rewrite and record its green against today's implementation as the
   baseline. This criterion locks existing behavior, so it has no red state by
   design; criteria 2/3's reds prove the suite exercises the mechanism.
2. **Ordering defined only in spec metadata; no site keys on literal "rename".**
   Author `test_pinned_tail_ordering_comes_from_spec_metadata_only`: override
   rename's registry entry with a spec that carries NO pinned metadata (plain
   `BuiltinSpec(validate, execute, singleton=True)`) and assert rename then
   sorts purely by stored position at all three consumers. RED today: every
   site keys on the literal string "rename", so rename stays pinned. GREEN
   after: the sites consult the (overridden) spec, which is unpinned.
3. **A second pinned type registered in a test sorts between positional steps
   and rename at all three consumers, no site-specific changes.**
   Author `test_second_pinned_type_sorts_between_positional_and_rename`:
   register `_pin_probe` with `pinned_tail=1` (identity executor), attach
   rename (pos 0), `_pin_probe` (pos 1), filter (pos 2); expect
   [filter, _pin_probe, rename] at get_pipeline, get_unified_pipeline, and
   run_pipeline. RED today: `BuiltinSpec` has no `pinned_tail` field — the
   registration surface the criterion requires does not exist (TypeError on
   construction is the missing behavior, not an incidental failure). GREEN
   after registration-only.
   Criteria 2 and 3 are satisfied by the same single mechanism, so both red
   tests are authored before the one implementation step and both greens land
   with it.
