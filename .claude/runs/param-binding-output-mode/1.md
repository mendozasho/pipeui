# Slice 1 — numeric-column-binding
Issue: #99 (sub-issues #101 backend, #102 UI)   Branch: feature/param-binding-output-mode-1

## Plan (written before coding)
- **Approach per layer:**
  - **business_logic (#101, climb data→business):**
    - `classification.py`: add ONE pure leaf helper `binding_kind(type_str)` derived
      off the EXISTING `function_class` — `scalar` → `"value_or_column"`,
      `pd.Series` → `"column_only"`, `pd.dataframe` → `"table"`. DB-free. Built off
      `derive_function_class([type_str])` so a single param type maps to its class,
      then class→kind. Unknown type raises (callers gate with `is_known_param_type`).
    - `suggest.py`: DELETE `_SUGGEST_TYPES` / `_SCALAR_TYPES`. Derive eligibility from
      `binding_kind`: a param is included unless its kind is `"table"` (pd.DataFrame).
      `param_kind` = `"column"` when kind is `column_only`, else `"scalar"` (numeric/str
      text). Emit a new `binding_kind` field per param. `value_or_column` numerics get
      `suggested_columns` + `current_bindings` (alias_map.position order) exactly like str.
    - `attach.py`: DELETE `_REQUIRES_BINDING`. Generalize `_find_missing_bindings`: a
      param needs a binding ONLY when `binding_kind == "column_only"` (pd.Series) OR it
      is `value_or_column` with NO literal AND NO has_default; satisfied by binding OR
      literal OR has_default. `_write_alias_rows`: write alias rows for any param that
      HAS a binding (not gated on a literal type set) — so column-bound numerics persist.
      Add `has_default` to the param_rows query so the rule can read it. The existing
      equal-length-among-varying guard (`_validate_bundles`/`pair_bundles`) already
      covers numeric varying columns — no change needed there.
  - **ui (#102):** `screen-builder.jsx` `PendingStepCard`: generalize the str-only
    conditionals to the backend-emitted `binding_kind` via a SINGLE predicate
    (`isValueOrColumn(p)` = `p.binding_kind === "value_or_column"`; `isColumnOnly` =
    `"column_only"`). Reuse the EXISTING `strModes`/`isColumnMode`/`handleSave`
    machinery, renamed conceptually to "toggle modes" but keep `strModes` state keyed by
    param_id for ALL value_or_column params (not just str). text→column clears the stale
    scalar (mirror existing str behavior). Save-guard: a value_or_column param with no
    value, no column, and no `has_default` blocks Save (mirror backend). replace-target
    picker already iterates `params.filter(isColumnMode)` — once a numeric is column-bound
    it appears automatically.
- **Utilities/patterns reused:**
  - `classification.derive_function_class` (class derivation), `is_known_param_type`.
  - attach: `pair_bundles`/`BundleLengthError` (equal-length guard), `_write_alias_rows`,

  - suggest: existing unified param loop, `current_bindings` ORDER BY am.position.
  - frontend: `strModes`, `isColumnMode`, `handleSave`, `initScalarValues`,

## Actual (written before push)
- What was built: A single `binding_kind(type_str)` leaf in classification.py derived off
  the existing `function_class`; suggest.py and attach.py had their parallel type literals
  (`_SUGGEST_TYPES`/`_SCALAR_TYPES`/`_REQUIRES_BINDING`) DELETED and now derive eligibility
  from `binding_kind` (suggest emits the field; attach's `_find_missing_bindings` generalized
  to binding-OR-literal-OR-`has_default` and `_write_alias_rows` writes rows for any bound
  param incl. numerics). The screen-builder modal's str-only toggle was generalized to all
  `value_or_column` params via one predicate set (`isValueOrColumn`/`isColumnOnly`/
  `isTableParam`) — numerics get the Plain string / Column-backed toggle, column mode clears
  the stale scalar, edit-load restores mode + column order, the Save-guard mirrors the backend
  block-at-attach, and the replace-target picker lights up once a numeric is column-bound.

- Deviations from plan (and why):
  - The `str` param's `param_kind` legitimately flips from "column" to "scalar" under the
    new rule (str is `value_or_column` — opens as free-text, toggles to column). This is the
    PRD's stated rule, but it broke three classes of EXISTING tests that encoded the OLD
    permissive behavior; all were updated to the new rule (per the PRD: "the existing
    test_scalar_param_exempt_from_binding / test_dataframe_param_exempt_from_binding are
    updated to the new rule"):
      * `test_attach.py::test_scalar_param_exempt_from_binding` + 3 attach-based scalar tests
        (`test_patch_scalar_values_*`, `test_get_pipeline_scalar_value_none_when_unset`) — a
        bare int param with no binding/literal/default now correctly FAILS to attach; updated
        to declare a Python default so the attach is valid while preserving each test's intent.
      * `test_attach.py::test_suggest_column_params_have_kind_column` — str is now param_kind
        "scalar" / binding_kind "value_or_column"; pd.Series is "column" / "column_only".
      * `screen-builder.test.jsx::"onSave sends a blank scalar..."` — a bare blank numeric is
        now blocked by the Save-guard; gave it a default so the blank still flows on save.
- Unanticipated files (in actual but not planned or issue touched_files):
  - `tests/test_api_pipelines.py` — slice 2's file (slice 2 depends_on:[1], NOT in-flight this
    single-slice round, so no conflict). One assertion (`test_dry_run_includes_param_kind_and_
    available_columns`) asserted str→param_kind "column"; the deliberate str rule change made it
    "scalar". Updated to the new rule + asserts the new `binding_kind` field, to keep the
    accumulator green for the integrator's full-suite gate. Phase-3 signal: slice 1's str
    rule change ripples into a slice-2-owned test file.
