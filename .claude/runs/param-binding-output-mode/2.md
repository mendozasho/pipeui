# Build record ‚Äî slice 2: per-function-output-mode

Feature: `param-binding-output-mode` ¬∑ Slice 2 ¬∑ branch `feature/param-binding-output-mode-2`
Parent issue #100; sub-issues #103 (backend/API, acceptance 0‚Äì3), #104 (UI, acceptance 4).

## Intent

Let a user set Append vs Replace independently for each function within a set in the
Report Builder. The per-function storage + execution already exist (#264):
`function_output_config` and `output_target_map`, both keyed by
`(source_function_map_id, function_id)`, and the runner / step_loader already read
per-function config first with a set-level fallback. This slice surfaces that through
the edit path (`patch_pipeline_step`), widens the existing step PATCH route to carry it,
and relocates the Builder's Append/Replace control from the set-level card into each
function group.

## Guardrails (from PRD ‚Äî non-negotiable)

- NO new tables, NO schema migration, NO runner/executor change. The storage +
  execution are already in place.
- Extend `patch_pipeline_step` with an OPTIONAL `function_output` map
  (`function_id` ‚Üí `{output_mode, append_name, output_targets}`) updating
  `function_output_config` + `output_target_map` keyed by `(sfm_id, function_id)`
  transactionally. When `function_output` is provided, STOP writing the vestigial
  set-level `source_function_map.output_mode`. `step_edit` stays the single owner.
- Widen the EXISTING `PATCH /pipelines/{source_id}/steps/{sfm_id}` handler ‚Äî NO new
  route. The route stays a thin seam: no raw SQL in `middleware/pipelines.py`
  (guarded by `test_source_read.py::test_middleware_seam_has_no_raw_sql`).
- UI builds ON TOP of slice 1's numeric-binding work in `screen-builder.jsx`; do not
  regress it. Relocate Append/Replace into each function's group with per-function
  append-name / ordered replace-target inputs, PATCH scoped to one `function_id`.

## Plan (red‚Üígreen‚Üírefactor per acceptance criterion)

### Backend / API (#103, acceptance 0‚Äì3) ‚Üí `tests/test_api_pipelines.py`

- [0] PATCH with a `function_output` map persists `output_mode`/`append_name` to
  `function_output_config` for the named `function_id` ONLY (siblings untouched);
  ordered replace targets persist to `output_target_map` keyed by `(sfm_id, function_id)`.
- [1] A set with one append + one replace member round-trips with independent
  per-member config.
- [2] With `function_output` given, set-level `source_function_map.output_mode` is NOT
  written; a legacy step with no per-function config still resolves via the existing
  fallback (step_loader reads `source_function_map.output_mode` when no config row).
- [3] An invalid `output_mode` or malformed `function_output` entry returns a
  STRUCTURED error (422), not a 500.

Implementation:
- `step_edit.patch_pipeline_step`: add `function_output: dict[uuid.UUID, dict] | None`.
  For each `(function_id, cfg)`: validate `output_mode` ‚àà {append, replace}; upsert
  `function_output_config (sfm_id, function_id)`; delete + re-insert `output_target_map`
  rows for `(sfm_id, function_id)` in `output_targets` order. All inside one
  BEGIN/COMMIT (ROLLBACK on error ‚Üí RuntimeError). When `function_output` is provided,
  the set-level `output_mode` UPDATE branch is suppressed.
- `middleware/pipelines.py`: add `function_output` to `PatchStepIn`; parse
  `{function_id_str: {output_mode, append_name?, output_targets?[col_id_str]}}` into
  `{UUID: {...with UUID targets...}}`; structured 422 on malformed UUIDs; delegate. No
  `.execute`.

### UI (#104, acceptance 4) ‚Üí `screen-builder.test.jsx`

- [4] Each function within a set renders its own Append/Replace control with its own
  append-name / replace-target inputs, and changing one issues a `function_output`
  PATCH scoped to that function only.

Implementation: relocate the set-level Output control out of `StepCard` into a
per-function control rendered inside each function's group. Each control reads the
function's `output_mode` / `append_name` / `output_targets` (already on `step.functions[i]`
from step_loader), and on change PATCHes
`{ function_output: { [fn.function_id]: { output_mode, append_name, output_targets } } }`.

### Note on an unanticipated file
`pipeline_read.py` was NOT in the slice's predicted `touched_files`, but the canvas
read path (`get_pipeline`) did not emit per-function output config ‚Äî only the runner's
`step_loader` did. Without extending the read seam the UI could not render or
round-trip the persisted per-function mode/append-name/targets (Principle 7). The
change is additive (new fields on each function in the GET response) and guarded by
`test_get_pipeline_emits_per_function_output_after_patch`. No new table, no new route,
no runner/executor change ‚Äî guardrails intact.
