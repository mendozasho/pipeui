# ADR-0002: Cross-feature decision memory is retrieval over frozen ledgers — no decisions registry

- **Status:** Accepted
- **Date:** 2026-06-12
- **Scope:** Pipeline tooling (the `grill-discovery` skill), not any single product feature.

> Like ADR-0001, this is a *tooling* ADR. It exists so that a future editor who notices
> decisions lack a harvested registry (the way file predictions have `.claude/hub-registry.json`)
> sees that the "missing" registry was deliberately rejected, not overlooked.

## Context

Resolved decisions live per-feature in frozen discovery ledgers (`.claude/discovery/<slug>.json`).
Until this decision, nothing read them across features: the grill's prior-art lens searched
*code*, so feature B could invisibly re-litigate a question feature A had settled. The obvious
fix — a global decisions registry fed by a `harvest-decisions` sibling of `harvest-hubs` —
carries hidden costs: graduation criteria (what is durable knowledge vs feature-local trivia),
a staleness/supersession doctrine, a new single writer, and a second system of record competing
with both the ledgers and the ADR tier.

## Decision

**The grill retrieves prior decisions directly from existing ledgers; no registry, index, or
harvester is created.** Concretely:

1. **The harvest-hubs analogy fails, so don't copy it.** A hub is a *derived* cross-feature
   aggregate (`slice_count >= threshold`) that no single build record holds — a registry is
   necessary there. A decision is directly reusable from its frozen ledger with question,
   resolution, and provenance intact. The gap was that nothing *looked*, not that data was missing.
2. **Retrieval runs at two points in the grill:** a one-shot grounding sweep keyed off the
   provocative restatement's terms (catches scope-level collisions early), and a per-question
   extension of the prior-art lens (catches mid-session re-litigation).
3. **Token frugality is a read discipline, not an artifact:** stage 1 extracts only the short
   `question` strings (and `explicitly_out_of_scope` entries) across ledgers, formatting-tolerantly
   (`jq`, not line-grep); stage 2 reads a full `resolution` only for plausible hits. The questions
   *are* the index.
4. **Hits are advisory with provenance, never auto-adopted.** Conflicting resolutions are all
   surfaced, most recent recommended, the user adjudicates; non-frozen ledgers' hits are flagged
   provisional. Adoption or divergence is restated in the *current* feature's ledger naming its
   source — frozen ledgers are never mutated, so there is no supersession machinery
   (same cost-asymmetry argument as ADR-0001: a stale surfaced decision costs one wasted
   interactive question; a missed one costs exactly the pre-feature status quo).

The ADR tier is untouched: ADRs stay the human-facing record for hard-to-reverse trade-offs;
ledgers stay the machine-grounding memory. An ADR's existence does not exempt its decision
from retrieval.

## When to revisit

- **Question-stage output becomes noisy in practice** (many dozens of ledgers): add a derived
  pointer index (slugs + question titles, never resolutions) regenerated by `grill-discovery`
  at freeze — a rebuildable cache atop the ledgers, exactly as `hub-registry.json` sits atop
  build records. The trigger is observed noise, not a file count picked in advance.
- **Ledgers stop being the durable source of decisions** (e.g. a future phase prunes or archives
  them): retrieval loses its corpus and a registry becomes a preservation mechanism, not a
  duplicate — reopen then.
