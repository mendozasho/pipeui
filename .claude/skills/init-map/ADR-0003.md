# ADR-0003: Shadowed-skill muting lives in init-map (model-judged, confirm-gated), not the installer

- **Status:** Accepted
- **Date:** 2026-06-13
- **Scope:** Pipeline tooling (the `init-map` skill and `install.sh`), not any single product feature.

> A tooling ADR, like ADR-0001/0002. It exists so a future editor who sees the installer
> already *detect* orphaned skills wonders why it doesn't just *mute* them too — and finds
> the split was deliberate, not an omission.

## Context

Installing the pipeline into a repo that already has its own skills can collide on names
(handled by `install.sh`'s overwrite flow) and, more subtly, leave the adopter's skills
*shadowing* a phase: two skills advertising the same intent (a project's own grilling skill
vs `grill-discovery`, a wave-runner vs `to-code`), so Claude Code's description-based
auto-selection can fire the wrong one. The installer can cheaply and reliably *detect*
orphans — a set difference against the seven shipped names — and does. But "does orphan X
genuinely overlap phase Y?" is a judgment about meaning, and "disable it" is a write to a
skill the installer never shipped.

## Decision

**Detection is the installer's; the overlap judgment and the mute are `init-map`'s, gated on
the adopter's confirmation.**

1. **The installer detects and reports, never mutes.** After a successful install it names the
   orphans and points at the lever, staying read-only on anything outside its Ship-list — its
   load-bearing principle is that it writes only what it ships.
2. **Overlap is judged by the model, in `init-map`'s adoption pass.** A shell installer can only
   keyword-match, which mis-mutes on shared words ("plan", "implement") and misses reworded
   overlaps. `init-map` already reads the repo and reports divergences without auto-rewriting;
   the shadow audit is the same shape, one tier stronger — it may close the gap, but only on a yes.
3. **The mute is `disable-model-invocation: true` on the orphan's frontmatter** — Claude stops
   auto-selecting it while `/<name>` still works, so it is non-destructive and one reversible
   line. Phases are never muted; non-overlapping orphans are never touched.
4. **The same split governs stale doc references.** An overwrite changes what a skill *means*
   without changing its name, so a router or workflow doc that referenced it can silently go
   stale. The installer flags that an overwrite happened (cheap detection); `init-map` judges
   which references actually went stale — comparing each doc's use against the installed
   contract — and surfaces them, never rewriting the router. Same boundary, same reason:
   judging meaning and touching the adopter's docs are the model's job, gated on the adopter.

## Considered options

- **Installer auto-mutes by keyword heuristic** — rejected: unreliable meaning-matching, and it
  breaks the installer's copy-only / Ship-list boundary by editing skills it didn't ship.
- **A new eighth skill dedicated to the audit** — rejected: the repo's identity is seven portable
  phases; the audit is an adoption-time duty `init-map` already has the repo context for.

## When to revisit

If a native per-skill enable/deny surfaces in Claude Code settings (today there is none — the
lever is frontmatter only), prefer it over editing each skill's frontmatter.
