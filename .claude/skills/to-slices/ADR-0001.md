# ADR-0001: The hub-file prior is advisory — bias it for safe failure, not precision

- **Status:** Accepted
- **Date:** 2026-06-11
- **Scope:** Pipeline tooling (the `to-slices` / `to-code` / `harvest-hubs` skills), not any single product feature.
- **Conform to:** the project's `ADR-FORMAT.md`; renumber to fit the existing ADR sequence.

> This is a *tooling* ADR — it records a decision about how the slicing pipeline predicts file overlap, not a decision about a feature being built. It exists so that a future editor (human or agent) who reaches for an "obvious improvement" to the hub registry sees the trade-off that the code alone does not show.

## Context

The pipeline serializes same-file work so that parallel slices don't collide. Correctness of that serialization is guaranteed by two **authoritative** mechanisms:

1. `to-code` integrates slices **sequentially**, rebasing each branch onto the accumulator tip before merge — so any overlap that slips through surfaces as a 2-way conflict against a known-good base, never an N-way pile-up.
2. `to-slices` step 2 **reactively** corrects a slice's predicted `touched_files` from the build-record `actual_touched_files` whenever `to-code` kicks a conflict back.

On top of these sits an **advisory** prior: `.claude/hub-registry.json`, maintained by `harvest-hubs` from past features' actuals, which `to-slices` reads to pre-seed `touched_files` with known hub files *before* any conflict reveals them. The prior's job is to reduce how often the reactive mechanisms have to fire — it is an optimization, not a correctness mechanism.

The temptation, on encountering the prior, is to make it *precise*: add decay so stale hubs age out, tighten hub-to-slice matching so slices only get the files they truly touch. Both "improvements" optimize the wrong objective and are recorded here as deliberately rejected defaults.

## Decision

**Treat the hub prior as advisory and optimize it for simplicity and safe failure, pushing any precision that correctness needs into the authoritative mechanisms above — never into the prior.** Concretely:

### 1. Accumulate without decay (default)

`harvest-hubs` only ever increments counts; a file that crossed the hub threshold stays a hub even if its area later goes quiet. Hubs are overwhelmingly *structural* (route registries, `__init__.py`, schema modules, manifests, shared fixtures), so staleness is rare and the cost of a stale hub is only mild over-serialization, not a wrong result. Decay is omitted to keep the harvester a pure accumulator and `to-slices` a pure reader.

### 2. Two knobs, deliberately pointing different ways

These look contradictory and are not — they govern different things:

- **`hub_threshold` (in `harvest-hubs`) guards signal quality.** A file needs real cross-feature evidence (default: 3 slices) before it is trusted as a hub, so the prior reflects structure, not coincidence. Conservative here means "don't promote noise."
- **Hub-to-slice matching (in `to-slices`) over-includes.** Once a file *is* a hub, a slice whose layers overlap the hub's recorded layers gets that file added to its `touched_files` unless there's a concrete reason it doesn't. Generous here means "don't miss a real hub."

The over-include at matching is justified by an **asymmetry of costs**: an *under*-predicted hub costs a real merge conflict (integrator toil + a kickback round-trip), while an *over*-predicted hub costs only that two otherwise-parallel slices run sequentially (wall-clock, no toil, no correctness risk). When the costs are asymmetric, bias toward the cheap failure.

## Consequences

- The prior is robust to being wrong: every miss is caught by sequential integration and corrected by the reactive loop, so the prior can be coarse without endangering correctness.
- Some pairs serialize unnecessarily. Accepted — it trades cheap wall-clock for avoided conflicts.
- Stale hubs persist. Accepted for structural-hub codebases.
- `to-slices` stays a pure reader of the registry; all learning logic lives in the single-writer `harvest-hubs`. This separation is load-bearing — it's what lets the registry evolve (e.g. gain decay) without touching the slicer.

## When to revisit

Reopen this decision only if one of its premises changes:

- **Merge conflicts become cheap** (e.g. reliable automatic resolution of hub collisions): the over-include bias weakens — exact matching becomes worth it, since the failure it avoids is no longer expensive.
- **Parallelism becomes scarce or expensive** (few agents, costly runtime): unnecessary serialization starts to dominate; lean toward a higher threshold and tighter matching.
- **The codebase churns structurally** (hubs move around release to release): add **decay in the `harvest-hubs` fold step** — weight recent features more — never in `to-slices`.
- **The prior is mistaken for a correctness mechanism** (someone proposes blocking a freeze on it, or having `to-slices` write it): reject — correctness lives in `to-code`'s sequential integration and `to-slices` step 2, by design.
